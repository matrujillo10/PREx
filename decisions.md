## 2026-05-09 — Problem framing: AI writes most code, human review is bottleneck

**Decision:** Problem statement = "Humans are AI code reviewers as a key piece of their job. Review experience must be redesigned around that premise."
**Rationale:** Backed by DORA 2025 (review time +441%, no-review merges +31%, incidents/PR +242%), CodeRabbit study (AI PRs 75% more logic errors, 2.74x sec issues), 17M agent PRs/month on GitHub (Mar 2026, +325% in 6mo).
**Alternatives considered:** Focus on AI-as-reviewer alone (rejected — already crowded market); focus on author productivity (rejected — different problem).

---

## 2026-05-09 — Two reviewer personas

**Decision:** Design for two distinct personas with shared diff primitive but different chrome:
1. **AI-assisted author** doing pre-PR self-review.
2. **Peer reviewer** reviewing code generated by others (with AI).

**Rationale:** Cognitive science + 2025-26 practitioner discourse converge on this split:
- Self-review catches **intent**, fails at **what's on page** (familiarity blindness).
- Peer review catches **what's on page**, misses **intent** (no recall).
- AI-assisted author has *neither* familiarity blindness nor intent-recall advantage. Role inverted toward peer reviewer (Hashimoto).
- Same surface = wrong default → produces rubber-stamping.

**Alternatives considered:** Single unified surface (rejected — implicitly says cognitive jobs are same); 3+ personas including release manager (deferred — out of MVP scope).

## 2026-05-09 — Type-spine + orthogonal axes architecture

**Decision:** PR type chosen by author from a closed list; multiple orthogonal axes auto-detected and composed.

**Type spine (full vision, ~12 types):** feat, fix, refactor, perf, test, docs, chore (folds build/ci/style), deps, security, migration, config, prompt/skill.

**Orthogonal axes (8):** scope/area, size (XS-XXL), SemVer impact, risk/blast radius, reversibility (one-way/two-way), generated-vs-handwritten, author class (human/AI/bot), breaking marker.

**Rationale:** Conventional Commits' `chore`/`build`/`ci`/`style` split divides on SemVer footer, not on what reviewer actually looks at — fold them. `migration`/`security`/`deps`/`config` each demand distinct checklist — elevate them out of CC. (Source: PR taxonomy research, 2026-05-09.)

**Alternatives considered:** Pure Conventional Commits (rejected — types divide on wrong axis); free-form labels (rejected — no checklist binding possible).

---

## 2026-05-09 — MVP scope: feat, fix, chore/build/ci

**Decision:** First implementation targets exactly three types: **feat**, **fix**, and **chore** (folding build + ci + style).

**Rationale:**
- Covers majority of PR volume in typical repo.
- `chore` is the highest rubber-stamp class (74-92% accept rate per agentic study) — biggest leverage to add discipline cheaply.
- `feat` and `fix` are the two highest-burden-increase types per AI-vs-human comparisons (CodeRabbit, Apiiro).
- Defers harder-to-spec types (migration, security, prompt/skill) until we validate the manifest + checklist mechanic.

**Alternatives considered:** Start with migration + security only (rejected — too narrow, hard to dogfood); start with all 12 (rejected — premature complexity).

---

## 2026-05-09 — MVP wedge: inline type-conditional checklist on diff hunks

**Decision:** Single biggest unbuilt UX primitive = **type-conditional checklist rendered inline on diff hunks**, not in a separate markdown doc.

**Rationale:** Classification → routing → CI checks → AI reviewer specialization layers are mature. UI layer is laggard. GitHub Files Changed renders identical for 3-line typo and 12-file schema migration. No mainstream review tool re-orders panels, hides irrelevant signals, or surfaces type-specific checklists *inline* in the diff.

**Alternatives considered:** Reviewer-side triage UI for collapsing AI comments (deferred — second-order); spec-first review (deferred — bigger paradigm shift, harder MVP); risk-tiered auto-merge (deferred — regulatory exposure on day 1).

---

## 2026-05-09 — Manifest-driven config: `.review/types.yaml`

**Decision:** Type → globs → checklist → required reviewers → required checks → AI agent prompt all bound in a repo-level manifest file.

**Rationale:**
- Portability across tools (no vendor lock-in to one taxonomy).
- Auditable — platform team owns manifest, not PR author.
- Anti-gaming: PR author can't relabel `feat` as `chore` to skip checks because checklist binding is platform-controlled.

**Alternatives considered:** PR-author-set type with no manifest (rejected — gaming trivial); fully automated classifier (deferred — MVP starts with author choice + classifier as tiebreaker/suggestion).
